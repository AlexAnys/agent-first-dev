## Часть 1: Как устроены LLM (базовые понятия)

### Что такое LLM?

**LLM (Large Language Model)** — это модель машинного обучения, обученная на огромном количестве текстов. Она умеет предсказывать, какое слово (или токен) идёт следующим в тексте.

### Как модель "думает"

```
Вы пишете: "Столица Франции — это..."
Модель думает: "С вероятностью 95% следующее слово — 'Париж'"
```

LLM не "знает" факты в человеческом понимании — она **предсказывает наиболее вероятное продолжение** на основе паттернов из обучающих данных.

### Что такое контекст (context window)?

**Контекст** — это всё, что модель "видит" при генерации ответа:
- Ваш системный промпт
- История диалога
- Текущий вопрос

Контекст ограничен (например, 4K, 8K, 128K токенов). Если текст слишком длинный — старые сообщения "забываются".

## Часть 2: Параметр Temperature

**Temperature** — ключевой параметр, влияющий на "креативность" модели.

| Temperature | Поведение | Когда использовать |
|-------------|-----------|-------------------|
| **0.0** | Детерминированный, всегда одинаковый ответ | Генерация кода, точные вычисления |
| **0.2-0.3** | Консистентный, с минимальными вариациями | Логические рассуждения, анализ |
| **0.5** | Умеренный баланс | Общие задачи |
| **1.0** | Максимально разнообразный | Когда нужны разные варианты |

**Правило:** Чем точнее нужен ответ — тем ниже temperature.

## Часть 3: Техники промптинга

### 1. K-Shot Prompting (Few-Shot Learning)

**Суть:** Показать модели несколько примеров "вход → выход", чтобы она поняла паттерн.

**Когда использовать:** Когда модель плохо понимает задачу с первого раза.

**Пример:**
```
Переверни буквы в слове:

hello → olleh
world → dlrow
python → nohtyp

Теперь переверни: computer → ?
```

**Почему работает:** Модель видит паттерн и применяет его к новому примеру.

**Совет из курса:** Используйте 3-5 примеров. Больше — не всегда лучше, может забить контекст.

---

### 2. Chain-of-Thought (CoT) — Цепочка рассуждений

**Суть:** Попросить модель рассуждать пошагово, а не давать ответ сразу.

**Когда использовать:** Математика, логические задачи, сложные рассуждения.

**Плохой промпт:**
```
Что такое 3^12345 mod 100?
```

**Хороший промпт (CoT):**
```
Реши эту задачу шаг за шагом. Покажи свои рассуждения.
В конце напиши ответ в формате "Answer: <число>"

Что такое 3^12345 mod 100?
```

**Почему работает:** Модель "проговаривает" промежуточные шаги, снижая вероятность ошибки. Это как когда человек решает задачу вслух — меньше шансов ошибиться.

**Магические фразы для CoT:**
- "Let's think step by step"
- "Объясни свои рассуждения"
- "Покажи промежуточные шаги"

---

### 3. Self-Consistency Prompting (Самосогласованность)

**Суть:** Запустить модель несколько раз с **высоким temperature** и выбрать наиболее частый ответ (голосование).

**Когда использовать:** Когда одиночный ответ ненадёжен.

**Алгоритм:**
```
1. Задать один и тот же вопрос 5 раз (temperature = 1.0)
2. Получить 5 разных ответов
3. Выбрать ответ, который встречается чаще всего

Ответы: [25, 25, 30, 25, 25] → Итоговый ответ: 25
```

**Почему работает:** Если модель приходит к одному ответу разными путями — скорее всего, он правильный.

**Важно:** Используется temperature = 1.0, чтобы получить разнообразие!

---

### 4. RAG (Retrieval-Augmented Generation)

**Суть:** Дать модели внешний контекст (документацию, данные), чтобы она генерировала ответ на их основе.

**Когда использовать:**
- Работа с документацией API
- Ответы на вопросы по конкретным документам
- Когда нужна актуальная информация

**Пример промпта:**
```
Контекст (используй ТОЛЬКО эту информацию):
---
API Reference
Base URL: https://api.example.com/v1
Authentication: Header X-API-Key: <your key>
GET /users/{id} — Returns {"id": <string>, "name": <string>}
---

Задача: Напиши функцию fetch_user_name(user_id, api_key)
которая получает имя пользователя по ID.
```

**Почему работает:** Модель не галлюцинирует — она опирается на предоставленные факты.

**Используется temperature = 0.0** для максимальной точности.

---

### 5. Tool Calling (Вызов инструментов)

**Суть:** Научить модель генерировать структурированный JSON для вызова функций.

**Когда использовать:** Когда нужно интегрировать LLM с внешними системами.

**Пример системного промпта:**
```
Ты ассистент с доступом к инструменту:

Инструмент: get_weather
Параметры: {"city": "<название города>"}
Описание: Возвращает погоду в городе

Когда нужно узнать погоду, верни JSON:
{"tool": "get_weather", "args": {"city": "..."}}
```

**Что генерирует модель:**
```json
{"tool": "get_weather", "args": {"city": "Moscow"}}
```

**Важно:**
- Чётко опишите формат вывода
- Перечислите все доступные инструменты
- Используйте низкий temperature (0.2-0.3)

---

### 6. Reflexion (Рефлексия / Итеративное улучшение)

**Суть:** Дать модели обратную связь о её ошибках и попросить исправить.

**Когда использовать:** Генерация кода, когда первая версия не работает.

**Алгоритм:**
```
1. Модель генерирует код
2. Код запускается и падает на тестах
3. Модели показываются ошибки:
   "Твой код упал на тесте: input='Password1' → expected True, got False
    Причина: не проверяется наличие заглавной буквы"
4. Модель генерирует исправленную версию
```

**Пример рефлексивного промпта:**
```
Твой предыдущий код:
```python
def is_valid_password(password):
    return len(password) >= 8
```

Ошибки при тестировании:
- Input: "Password1!" → expected True, got True ✓
- Input: "password1!" → expected False, got True ✗ (нет проверки uppercase)

Исправь код, учитывая эти ошибки.
```

**Почему работает:** Модель "учится" на своих ошибках прямо в диалоге.

## Часть 4: Структура хорошего промпта

### Анатомия промпта

```
┌─────────────────────────────────────┐
│  СИСТЕМНЫЙ ПРОМПТ                   │  ← Роль и правила поведения
│  "Ты опытный Python-разработчик"    │
├─────────────────────────────────────┤
│  КОНТЕКСТ (опционально)             │  ← Документы, примеры, данные
│  "API documentation: ..."           │
├─────────────────────────────────────┤
│  ЗАДАЧА                             │  ← Что нужно сделать
│  "Напиши функцию, которая..."       │
├─────────────────────────────────────┤
│  ФОРМАТ ВЫВОДА                      │  ← Как должен выглядеть ответ
│  "Ответ в формате JSON: {...}"      │
└─────────────────────────────────────┘
```

### Чек-лист хорошего промпта

- [ ] **Чёткая роль** — кто модель в этом диалоге
- [ ] **Конкретная задача** — что именно нужно сделать
- [ ] **Формат вывода** — как должен выглядеть ответ
- [ ] **Примеры** (если нужно) — несколько образцов вход-выход
- [ ] **Ограничения** — чего делать НЕ нужно

### Примеры плохих и хороших промптов

**Плохо:**
```
Напиши код
```

**Хорошо:**
```
Ты Python-разработчик.
Напиши функцию calculate_tax(income: float) -> float,
которая вычисляет налог по прогрессивной шкале:
- До 10000: 0%
- 10001-50000: 10%
- Более 50000: 20%

Верни только код в блоке ```python```, без объяснений.
```

## Часть 5: Практические советы

### Когда какую технику использовать

| Задача | Техника | Temperature |
|--------|---------|-------------|
| Простой вопрос-ответ | Обычный промпт | 0.3-0.5 |
| Модель не понимает формат | K-Shot (примеры) | 0.5 |
| Математика/логика | Chain-of-Thought | 0.2-0.3 |
| Критически важный ответ | Self-Consistency | 1.0 |
| Работа с документами | RAG | 0.0 |
| Интеграция с API | Tool Calling | 0.2-0.3 |
| Код с ошибками | Reflexion | 0.2 |

### Частые ошибки

1. **Слишком абстрактные инструкции**
   - Плохо: "Сделай хорошо"
   - Хорошо: "Проверь на наличие SQL-инъекций"

2. **Отсутствие формата вывода**
   - Плохо: "Дай ответ"
   - Хорошо: "Ответ в формате: Answer: <число>"

3. **Слишком много примеров**
   - 3-5 примеров обычно достаточно
   - Больше = меньше места для контекста

4. **Неправильный temperature**
   - Для кода: 0.0-0.2
   - Для креатива: 0.7-1.0

5. **Не указана роль модели**
   - Добавьте: "Ты эксперт в..."

## Часть 6: Резюме

### Главные принципы

1. **LLM предсказывает, а не "знает"** — всегда проверяйте факты
2. **Контекст ограничен** — давайте только релевантную информацию
3. **Temperature влияет на вариативность** — ниже = точнее, выше = креативнее
4. **Примеры помогают** — показывайте модели, что вы хотите получить
5. **Итерация работает** — давайте обратную связь и просите исправить

### Когда LLM работает хорошо

- Генерация текста по шаблону
- Объяснение кода
- Рефакторинг
- Написание тестов
- Работа с документацией

### Когда LLM работает плохо

- Точные вычисления (без калькулятора)
- Актуальные события (данные устаревают)
- Длинные логические цепочки
- Задачи, требующие "здравого смысла"
